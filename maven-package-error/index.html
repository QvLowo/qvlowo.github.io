【Maven】打包Spring Boot專案後jar檔啟動異常的問題排查紀錄

發表於2025-03-14|更新於2026-01-09|development-practice
|總字數:1.1k|閱讀時間:4分鐘|瀏覽量:

前言

在練習 Docker部署Spring Boot專案時，我發現容器一直無法啟動。
一開始以為 Docker設定有問題，後來懷疑可能是jar檔打包不完整。
結果一步步追查後，竟然發現是自己犯了一個超蠢的錯誤?!

測試jar檔

在打包前，我先用IDE啟動Spring Boot專案，本地端正常。
於是我打包成jar檔後，用指令直接執行看看：

結果報出和在 Docker 裡完全相同的錯誤訊息。

主控台顯示：

No active profile set, falling back to 1 default profile: >”default”
…
Error creating bean with name ‘xxxServiceImpl’ Unsatisfied dependency expressed through constructor parameter 0 …
Error creating bean with name ‘xxxServiceDao’ Unsatisfied dependency expressed through constructor parameter 0 …
defined in @EnableJpaRepositories declared on JpaRepositoriesRegistrar.EnableJpaRepositoriesConfiguration: Cannot resolve reference to > bean ‘jpaSharedEM_entityManagerFactory’ while setting bean property ‘entityManager’

Debug 過程

(步驟莫名看起來有點像AI Agent的推理流程，但真的不是，是我自己一步步查出來的ˋAˊ!!)

一、分析原因

拆成兩部份來看：

  1. 這段只是提醒目前使用預設的 application.properties

    一般可能會額外有application-dev.properties或application-prod.properties
    No active profile set, falling back to 1 default profile: >”default”

  2. 真正的錯誤發生在 Bean 無法正確建立：

    Error creating bean with name ‘xxxServiceImpl’ Unsatisfied dependency expressed through constructor parameter 0 …
    Error creating bean with name ‘xxxDao’ Unsatisfied dependency expressed through constructor parameter 0 …
    defined in @EnableJpaRepositories declared on JpaRepositoriesRegistrar.EnableJpaRepositoriesConfiguration: Cannot resolve reference to > bean ‘jpaSharedEM_entityManagerFactory’ while setting bean property ‘entityManager’

問題的根源很明顯：
SpringBoot無法正確注入某些 Bean，導致Service層依賴注入失敗。

二、推理與排查

於是我開始檢查所有可能出錯的地方：
✅ 各類別的@Component/@Service是否有加
✅ 注入有正確注入 (我使用Lombok + Constructor Injection)
✅ Lombok正常運作（編譯執行 不影響打包）
✅ @Entity標註正確，Table/Column 名稱無誤

那我就想奇怪了(?) 程式邏輯都沒問題，於是我懷疑是Maven打包設定出了問題。

執行 mvn clean package，最下面的BUILD SUCCESS讓我以為沒有問題，
結果往上滑一看：
dependency.scope' for com.mysql:mysql-connector-j:jar must be one of [provided, compile, runtime, test, system]

咦？scope？MySQL我明明只設定版本而已呀@A@?!

打開pom.xml一看，原來我誤把<version>寫成<scope>

修正方法

把<scope>改為<version>就可以了

笑死 小丑竟是我自己 555

這次的教訓：
不要太相信IDE能啟動就代表沒問題，
也不要看到”BUILD SUCCESS”就鬆懈。
仔細檢查每個步驟的輸出訊息，往往錯誤早就藏在裡面。

結語

這次的錯誤雖然只是小失誤，但讓我學到：

打包不是啟動成功就代表沒事。

遇到異常時，別急著懷疑框架或Docker，
從最基本的設定檔(如 pom.xml)檢查起，
往往能更快找出真正的問題所在。
