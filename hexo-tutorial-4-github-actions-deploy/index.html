Hexo Blog教學(四)：GitHub Actions自動部署教學從手動到全自動的完整流程

發表於2025-10-16|更新於2026-01-22|learning-guides
|總字數:1.4k|閱讀時間:5分鐘|瀏覽量:

前言

上一篇：將你的網站手動部署到 GitHub Pages

在上一篇文章中，我們學會了如何透過hexo的指令hexo clean、hexo generate、hexo deploy來手動部署Hexo Blog。
雖然流程不複雜，但每次更新文章都要重複輸入這些指令，對於長期經營blog來說，很麻煩而且隨著文章內容變多，產生靜態檔的時間也會變得很久。

這篇文章會介紹如何利用GitHub Actions來自動部署Hexo Blog。
只要把文章push到GitHub，系統就會自動完成建置與部署，不需要再手動輸入指令，大幅節省時間，這也是現在目前最常見也很穩定的做法。

什麼是 GitHub Actions？

GitHub Actions 是由Github提供CI/CD的一個服務，只要你在指定的資料夾內，建立workflow文件，它就能在偵測某些事件發生(例如：push)時，自動執行你寫的工作流程腳本。
同時，他也有一個很強大的功能是，你可以設定工作排程，在指定的時間自動執行腳本，我覺得這一點非常的棒。

拿這個系列教學文來說，在部署hexo blog時，我們可以設定一個workflow是：

  1. 當你完成文章並push到GitHub的遠端倉庫的main分支時，workflow被自動觸發。
  2. GitHub Actions根據你寫的腳本，自動執行指令，像是npm install、hexo g產生靜態檔案等指令。
  3. 最後將生成的靜態檔案部署到GitHub Pages。
    這樣你就不需再手動操作Hexo指令三件套，節省部署的步驟。

GitHub Action自動部署流程

我們會使用兩個repository，分別是：

  1. Hexo根目錄的Repository
  • 倉庫權限：Private
  • 用途：保存Hexo專案與工作流程設定
  • 設置：私鑰(Deploy Key)
  1. 靜態檔案輸出的Repository(username.github.io)
  • 倉庫權限：Public
  • 用途：部署Hexo生成的靜態網頁
  • 設置：公鑰，並允許寫入權限

第一步：建立你的部署金鑰(Deploy Key)

開啟Terminal，執行下列這段指令：

-t ed25519:指定金鑰演算法，這個ed25519演算法比RSA更新且安全。
-C “deploy-key-for-my-blog”:為金鑰加上註解，方便識別。
-f “deploy_key”: 指定輸出檔案名稱為deploy_key(私鑰)和deploy_key.pub(公鑰)

指令執行後，應該會產生兩個檔案：

  • deploy_key(私鑰)
  • deploy_key.pub(公鑰)

常見問題（FAQ）

Q1：GitHub Actions 無法部署成功怎麼辦？
可以檢查這些點：

  • DEPLOY_KEY是否設定正確，且私鑰設定的名稱與workflow腳本的名稱一致。
  • workflow的repository名稱是否正確。
  • .yml檔案縮排格式是否正確。

Q2：可以只用一個Repository部署嗎？
可以，但分開兩個Repo(Hexo原始碼與靜態檔案)會比較安全。
如果要只用一個Repository， - name: Push Pages to blog repository這段的內容就要修改，因為我們就不用從A repo 部署到B repo。
