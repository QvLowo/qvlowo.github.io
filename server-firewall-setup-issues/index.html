發表於2025-03-03|更新於2026-01-12|development-practice
|總字數:1.2k|閱讀時間:3分鐘|瀏覽量:

前言

公司在開發新專案時，需要部署程式碼到新的 Server 上。
申請的伺服器是全新的空機，要自己裝Java、Maven、Database等等。
裝完後一切看似正常，但卻出現了防火牆開通了，卻仍然被擋的奇怪現象。

這篇就紀錄一下這次架設Server因為防火牆而踩過的坑，希望能幫到之後遇到相同狀況的人。

開啟防火牆port

不論Server是Linux 還是 Windows，新環境通常都需要手動開放防火牆Port 才能讓本地端開發電腦成功連線。
例如在啟動Spring Boot專案時，若無法連上資料庫或API，大多就是防火牆設定的問題。

補充：跨網段防火牆的坑

不過事情當然沒那麼單純…這次我真正踩的坑其實是跨網段連線。

公司內部網段有防火牆保護政策，跨網段連線需要額外申請。
我雖然有申請，但實際上還是無法連線，結果發現是網管那邊某個 Port沒開。

小知識：同網段 vs. 跨網段

網段 生活情境 是否需要路由器
同一網段 像在同個社區裡，拿包裹給隔壁鄰居可以直接按電鈴。 不需要路由器，直接能互通
不同網段 像住不同地址，要透過物流中心轉送。 要透過路由器轉發，防火牆沒開會被擋
有些學習到的小知識，日常開發可能不是那麼必要，但某天可以拿來應用，我覺得蠻棒的OwOb

公司不給開 Port？搬出Nginx反向代理

有時公司基於資安政策不開放特定Port(像 8080)，
這時可以用Nginx反向代理(Reverse Proxy)轉發請求。

範例：

公司允許80 Port，但不開8080。
我在Ngin 設定讓 80 → 8080，這樣就能透過80 Port間接存取 API。

簡單來說，Nginx 負責擔任路由轉發員，
把流量從外部轉給內部正確的服務端口。
(詳細設定之後會再寫一篇紀錄)

結論

這次部署新專案學到的教訓：

防火牆不只是Server設定的問題，也可能是網段之間的防火牆在搞鬼QQ

不論是 Windows、Linux 還是企業內網環境，
遇到無法連線的狀況，建議依序檢查：

  1. 應用程式是否啟動
  2. 防火牆Port是否開啟
  3. 網段之間是否有額外限制
